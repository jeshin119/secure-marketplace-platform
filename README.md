## 1. 프로젝트 개요

### 1.1 프로젝트명
보안 강화 중고거래 플랫폼(Vintage Market) — 블랙박스 모의해킹 및 취약점진단

### 1.2 배경
- **보안 위협 고도화**: 지능적 APT(Advanced Persistent Threat) 공격이 증가하면서, 방어자 관점의 점검만으로는 실제 위협에 대응하기 어려운 상황임. 공격자 관점의 테스트(공격 표면 분석, 시나리오 기반 침투)가 요구됨
- **내부 점검의 한계**: 기존 코드/설정 위주의 내부 점검은 운영 환경에서 실제로 악용 가능한지 여부를 충분히 검증하지 못함. 외부 노출 자산을 기준으로 한 블랙박스 진단이 필요함
- **검증·침해 대응 역량 확보**: 위협을 빠르게 탐지하고 대응하기 위해 MITRE ATT&CK 기반의 공격 시나리오를 활용하여, 침해 발생 시 대응 절차를 사전에 점검할 필요가 있음

본 프로젝트는 실제 운영 가능한 E-commerce 서비스를 가상으로 구축하고, 고객사 Vintage Market의 의뢰를 받아 외부에서 내부망까지 이어지는 잠재적 침해 경로를 사전에 식별하고, 위험도에 따른 개선 방안을 수립하는 실전형 블랙박스 모의해킹 프로젝트임.

### 1.3 목적
- 고객사 Vintage Market 플랫폼(웹/모바일 백엔드, 데이터베이스, 사내망, CI/CD 인프라)에 대한 자산·데이터 흐름·네트워크 구성 파악
- DFD/STRIDE 기반 위협 모델링을 통한 잠재 위협 식별
- MITRE ATT&CK에 매핑된 침투 시나리오 수립 및 실제 수행(외부 → DMZ → 내부망)
- 발견된 취약점의 위험도 평가 및 우선순위 기반 개선 권고안 제시, 일부 항목에 대한 보안 강화 적용

## 2. 시스템 아키텍처 (진단 대상 시스템)

정보수집(자산 식별·네트워크 구성 파악) 단계에서 확인된 Vintage Market의 인프라 구조는 아래와 같이 DMZ / 내부망 / 개발망(CI·CD)으로 분리되어 있음.

```
                ┌───────────────────────────── DMZ (172.20.0.0/24) ─────────────────────────────┐
                │                                                                                 │
  외부 사용자  ──▶│  Frontend (React SPA, :5173)  ──▶  Backend API (Express + Socket.IO, :3001)     │
 (위협 행위자)   │                                                                                 │
                └───────────────────────────────────────────┬─────────────────────────────────────┘
                                                              │ (Backend는 내부망과도 연결되는 피벗 자산)
                ┌───────────────────────── 내부망 (172.21.0.0/24) ──────────────────────────────────┐
                │                                                                                 │
                │   Database (MySQL 8.0)   ◀──  관리자용 DB 콘솔(phpMyAdmin, :8081)                  │
                │                                                                                 │
                │   내부 업무망 호스트 (Ubuntu, SSH) — 운영 메모/내부 자산정보 보관                          │
                │                                                                                 │
                └─────────────────────────────────────────────────────────────────────────────────┘

                ┌───────────────────────── 개발망 (172.22.0.0/24) ───────────────────────────────────┐
                │                                                                                 │
                │   CI/CD 서버(Jenkins, :8080/:50000)  ◀── 소스 저장소(Gitea)                          │
                │   * Jenkins 컨테이너는 Docker 소켓(/var/run/docker.sock)에 접근 가능                   │
                │                                                                                 │
                └─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.1 자산 식별 결과
| 영역 | 자산 | 설명 |
|---|---|---|
| DMZ | Frontend (React SPA) | 외부 사용자가 직접 접근하는 웹 클라이언트 |
| DMZ ↔ 내부망 | Backend API (Express, Socket.IO) | REST API + 실시간 채팅 서버. DMZ와 내부망에 동시에 연결되어 있어, 침해 시 내부망 진입점이 될 수 있는 핵심 피벗 자산 |
| 내부망 | MySQL 데이터베이스 | 회원/상품/거래/채팅/커뮤니티 데이터를 저장하는 핵심 자산 |
| 내부망 | DB 관리 콘솔(phpMyAdmin) | 데이터베이스에 대한 GUI 관리 인터페이스 |
| 내부망 | 내부 업무용 호스트 | 임직원 PC를 가정한 내부망 호스트. SSH 활성화, 운영 관련 메모/자산정보 보관 |
| 개발망 | CI/CD 서버(Jenkins) + 소스 저장소(Gitea) | 빌드/배포 파이프라인. Docker 소켓 접근 권한을 보유해 컨테이너 인프라 전체에 영향을 줄 수 있는 고위험 자산 |

### 2.2 데이터 흐름(DFD) 개요
- **인증 흐름**: 사용자 → Frontend → `/api/auth/*` → DB 조회/JWT 발급 → 이후 모든 API 호출에 `Authorization: Bearer <token>` 첨부
- **거래 흐름**: 상품 등록/조회/구매 요청 → Backend → DB(상품/거래/쿠폰/사용자 크레딧) 갱신
- **실시간 채팅 흐름**: Frontend ↔ Socket.IO(Backend) ↔ DB(`chat_messages`, `chat_rooms`) — REST API와 별도 경로로 DB에 직접 접근
- **파일 흐름**: 사용자 업로드(상품 이미지, 게시글 첨부) → Backend 저장소(`/uploads`) → 정적 자원으로 재서빙
- **운영/배포 흐름**: 개발자 커밋 → Gitea → Jenkins 빌드 → Docker 이미지 생성/배포

## 3. 기술 스택 및 선정 이유

### 3.2 모의해킹 수행 방법론 및 도구 선정 이유
| 항목 | 선정 내용 | 선정 이유 |
|---|---|---|
| 진단 방식 | 블랙박스(Black-box) 테스트 | 고객사 내부 정보 없이 외부 공격자 시점에서 접근 가능한 자산만으로 침투를 시도, 실제 위협 시나리오에 가장 근접한 검증 결과 확보 |
| 위협 모델링 | DFD + STRIDE | 데이터 흐름도를 기반으로 Spoofing/Tampering/Repudiation/Information Disclosure/DoS/Elevation of Privilege 6개 범주로 위협을 체계적으로 도출, 누락 없는 위협 목록 작성 가능 |
| 침투 시나리오 매핑 | MITRE ATT&CK Framework | 정찰(Reconnaissance)부터 초기 접근(Initial Access), 권한 상승, 내부 이동(Lateral Movement), 영향(Impact)까지 표준화된 전술/기법으로 시나리오를 구성, 결과 리포트의 재현성과 커뮤니케이션 효율성 확보 |
| 테스트베드 구성 | Docker Compose 기반 격리 네트워크(DMZ/내부망/개발망 분리) | 고객사의 실제 네트워크 분리 구조(DMZ-내부망-개발망)를 동일하게 재현, 네트워크 경계를 넘는 횡적 이동(lateral movement) 시나리오까지 안전하게 검증 |
| 산출물 형식 | 위협 테이블 + 시나리오 문서 + 위험도 평가 리포트 | 기술팀(개선 작업)과 의사결정자(우선순위/예산) 모두에게 전달 가능한 이중 산출물 구조 |

## 4. 핵심 기능 (프로젝트 수행 프로세스)

본 프로젝트는 아래 4단계 프로세스로 수행되었으며, 각 단계의 산출물이 다음 단계의 입력으로 사용되는 구조임.

### 4.1 1단계 — 정보 수집
- **자산 식별**: Frontend/Backend/DB/관리 콘솔/내부 호스트/CI·CD 서버 등 전체 자산 목록화 (2.1절 참조)
- **데이터 흐름 파악**: 인증, 거래, 채팅, 파일 업로드/다운로드, 배포 파이프라인까지 DFD 작성
- **서비스 명세 분석**: REST API 엔드포인트(인증/상품/거래/채팅/커뮤니티/업로드/다운로드/관리자) 및 실시간 통신(Socket.IO) 명세 정리
- **네트워크 구성 및 보안 설정 파악**: DMZ/내부망/개발망 분리 여부, 포트 노출 현황(5173, 3001, 8081, 8080 등), CORS·세션·헤더 설정 등 보안 설정 점검

### 4.2 2단계 — 위협 분석 및 침투 시나리오 수행 (DMZ 영역)
- **STRIDE 위협 모델링**: DFD 상의 각 데이터 흐름/프로세스/저장소에 대해 STRIDE 6대 범주로 위협 항목 도출 및 위협 테이블 작성
- **침투 시나리오 작성**: 외부 공격자가 DMZ 자산(Frontend/Backend)을 통해 접근 가능한 공격 표면을 기준으로 시나리오 수립
  - 인증 우회 및 권한 탈취 시나리오
  - 입력값 처리 결함을 통한 데이터베이스/서버 측 명령 실행 시나리오
  - 파일 업로드/다운로드 기능을 통한 비인가 파일 접근 시나리오
  - 거래 로직(결제·재고)에 대한 비즈니스 로직 공격 시나리오
- **침투 시나리오 수행**: 실제 환경에서 시나리오를 실행하고 결과(성공/부분성공/실패)와 증적(요청/응답, 로그)을 기록

### 4.3 3단계 — 내부 침투
- **내부망 노드 식별**: DMZ 자산(Backend)을 경유해 접근 가능한 내부망 자산(DB, phpMyAdmin, 내부 업무 호스트) 식별
- **내부 침투 시나리오 수행**:
  - Backend를 피벗으로 한 내부망 데이터베이스 접근 시나리오
  - 내부 업무 호스트에서의 정보 수집(자격증명, 내부 인프라 정보 등) 및 추가 침투 가능성 검증
  - 개발망(CI/CD) 자산에 대한 접근 가능성 및 Docker 소켓을 통한 권한 확대 가능성 검증

### 4.4 4단계 — 결과 종합 및 개선 제안
- **침투 테스트 결과 요약**: 단계별 시나리오 수행 결과를 ATT&CK 전술/기법 단위로 정리
- **취약점 영향 분석**: 각 발견 항목에 대해 CVSS 기반 위험도(심각도/악용 난이도/영향 범위) 평가
- **개선 우선순위 제안**: 비즈니스 영향과 악용 난이도를 기준으로 Critical → High → Medium → Low 순으로 개선 권고안 제시

## 5. 주요 침투 시나리오

2단계(위협 분석 및 침투 시나리오 수행) 및 3단계(내부 침투)에서 실제로 수행된 시나리오 중, 비즈니스 영향이 가장 큰 3건을 ATT&CK 전술/기법 단위로 상세 기술함. 각 시나리오는 6.1절의 발견 항목 번호와 연계됨.

### 5.1 시나리오 1 — 결제 시스템의 동시성 처리 취약점 악용 (관련 발견 항목: #6)

**개요**: 외부 공격자가 정상 사용자 계정으로 인증된 상태에서, 결제(구매) 처리 로직의 동시성 제어 결함을 악용하여 금전적 이득을 취하는 시나리오임.

| 단계 | ATT&CK 전술/기법 | 수행 내용 |
|---|---|---|
| 1. 정찰 | TA0043 Reconnaissance (T1592 Gather Victim Host Information) | 일반 사용자 계정으로 로그인 후 구매 가능한 상품 목록, 가격, 본인의 보유 크레딧을 확인. 1회 구매 후 재고 상태(`isSold`)가 즉시 변경되는지 응답을 통해 관찰 |
| 2. 초기 접근 | TA0001 Initial Access (T1078 Valid Accounts) | 정상적으로 발급받은 인증 토큰으로 구매 API(`POST /api/products/:id/purchase`) 호출 권한을 확보 |
| 3. 실행 | TA0002 Execution (CWE-362 Concurrent Execution using Shared Resource) | 보유 크레딧으로 1회 구매 가능한 고가 상품에 대해, 동일한 구매 요청을 다중 스레드로 동시에 전송. 서버가 재고 확인과 크레딧 차감·재고 갱신을 하나의 트랜잭션·잠금 없이 순차 처리함에 따라, 복수의 요청이 동시에 "구매 가능" 상태를 통과함 |
| 4. 영향 | TA0040 Impact (데이터 변조 — 자산/잔액 정합성 훼손) | 동일 상품에 대해 복수의 거래(Transaction) 레코드가 생성되고, 보유 크레딧 이상의 금액이 차감되어 잔액이 비정상적으로 변동됨. 동일 재고가 여러 구매자에게 "구매 완료"로 처리됨 |

**결과**: 공격자는 실제 보유 자산을 초과하는 금액의 상품을 취득할 수 있으며, 단일 재고에 대해 복수의 구매 확정이 발생해 판매자·플랫폼 측에 정산 손실이 발생함. 동시성 결함은 자동화된 동시 요청 스크립트만으로 재현 가능하여 악용 난이도가 낮음.

### 5.2 시나리오 2 — 웹 애플리케이션 입력 검증 우회를 통한 데이터베이스 접근 (관련 발견 항목: #1, #8)

**개요**: 공격자가 웹 애플리케이션의 사용자 입력 처리 과정에서 발생하는 검증 우회를 통해 인증을 무력화하고, 데이터베이스 내 민감 정보(회원 계정, 권한 정보)에 직접 접근하는 시나리오임.

| 단계 | ATT&CK 전술/기법 | 수행 내용 |
|---|---|---|
| 1. 정찰 | TA0043 Reconnaissance (T1595 Active Scanning) | 로그인, 검색, 채팅 등 사용자 입력을 받는 엔드포인트를 식별. 에러 응답에 포함된 상세 메시지·스택 트레이스(발견 항목 #8)를 통해 백엔드가 Sequelize/MySQL 조합으로 동작함을 확인 |
| 2. 초기 접근 | TA0001 Initial Access (T1190 Exploit Public-Facing Application) | 로그인 API의 이메일 파라미터에 SQL 구문을 삽입하여 인증 조건을 변조, 비밀번호 검증을 우회한 로그인 성공 여부를 검증 |
| 3. 자격 증명 접근/수집 | TA0006 Credential Access, TA0009 Collection (T1552 Unsecured Credentials, T1005 Data from Information Repositories) | 실시간 채팅 메시지 저장 쿼리에서 입력값이 검증 없이 INSERT 쿼리에 문자열로 결합되는 구조를 이용해 추가 쿼리 구문을 주입, 회원 테이블의 이메일·비밀번호 해시·권한(역할) 정보를 응답 또는 채팅 메시지 형태로 추출 |
| 4. 영향 | TA0008 Lateral Movement으로의 연결 가능성 (T1078 Valid Accounts) | 탈취한 관리자 계정 자격 증명을 이용해 관리자 페이지 및 관리 기능에 대한 비인가 접근 가능성 확보 |

**결과**: 전체 회원 계정 정보(이메일, 비밀번호 해시, 권한 정보 포함)가 외부로 유출될 수 있으며, 탈취한 자격 증명을 통해 관리자 권한 탈취로 이어질 가능성이 확인됨.

### 5.3 시나리오 3 — 웹 서버 취약점을 통한 RCE 및 내부망 침투 (관련 발견 항목: #3, #11, #12)

**개요**: 공격자가 웹 서버의 콘텐츠 처리 및 파일 업로드 기능의 보안 약점을 연계하여 원격 코드 실행(RCE) 권한을 확보하고, 이를 기점으로 DMZ에서 내부망 및 개발망(CI/CD)까지 침투하는 시나리오임. 본 시나리오는 3단계(내부 침투) 결과와 직접 연계됨.

| 단계 | ATT&CK 전술/기법 | 수행 내용 |
|---|---|---|
| 1. 정찰 | TA0043 Reconnaissance (T1595 Active Scanning) | 상품/게시글 첨부파일 업로드 기능과 업로드된 파일이 `/uploads/` 경로로 직접 서빙되는 구조를 확인. 파일 검증 로직이 파일명에 허용 확장자 문자열을 포함하는지만 확인하는 점을 식별 |
| 2. 초기 접근 | TA0001 Initial Access (T1190 Exploit Public-Facing Application) | 파일명에 허용 확장자와 실행 가능한 확장자(.jsp 등)를 함께 포함시켜 검증 로직을 우회한 뒤 업로드 |
| 3. 실행 | TA0002 Execution (T1505.003 Server Software Component: Web Shell, T1059 Command and Scripting Interpreter) | 업로드된 파일에 명령 실행을 위한 파라미터를 추가하여 접근, 웹 서버(Backend 컨테이너) 상에서 임의 OS 명령 실행 권한을 확보 |
| 4. 내부 탐색 | TA0007 Discovery (T1082 System Information Discovery, T1018 Remote System Discovery) | 확보한 명령 실행 권한으로 컨테이너의 환경변수·네트워크 설정을 조회, Backend가 DMZ와 내부망에 동시에 연결된 피벗 구조임을 확인하고 내부망 대역을 스캔하여 DB·phpMyAdmin·내부 업무 호스트를 식별 |
| 5. 내부 이동 | TA0008 Lateral Movement (T1021 Remote Services, T1552 Unsecured Credentials) | 내부 업무 호스트에 접근하여 평문 자격 증명 및 내부 인프라 정보가 포함된 메모 파일을 확보, 해당 정보를 이용해 개발망의 CI/CD(Jenkins/Gitea)에 접근 |
| 6. 권한 확대/영향 | TA0004 Privilege Escalation, TA0040 Impact (T1611 Escape to Host, T1610 Deploy Container) | Jenkins 컨테이너가 보유한 Docker 소켓 접근 권한을 이용해 컨테이너를 탈출, 호스트 수준 권한을 확보하여 전체 서비스 인프라에 대한 통제권을 확보 |

**결과**: 외부에서 접근 가능한 단일 웹 서버 취약점(파일 업로드 검증 미흡)을 시작점으로, DMZ → 내부망 → 개발망(CI/CD)에 이르는 전체 침해 경로가 실제로 성립함이 확인됨. 이는 본 진단에서 식별된 가장 심각한 침해 경로(Critical)에 해당함.

### 5.4 시나리오 4 — 내·외부 ARP Spoofing을 통한 phpMyAdmin 접근 (관련 발견 항목: #13)

**개요**: 공격자가 내부망(또는 내부망과 인접한 네트워크 구간)에서 ARP Spoofing을 이용해 트래픽을 가로채고, 별도 인증 절차가 없는 DB 관리 콘솔(phpMyAdmin)에 접근하는 시나리오임.

| 단계 | ATT&CK 전술/기법 | 수행 내용 |
|---|---|---|
| 1. 정찰 | TA0043 Reconnaissance (T1595 Active Scanning, T1040 Network Sniffing) | 내부망 대역의 활성 호스트 및 ARP 테이블 구성을 스캔하고, DB 관리 콘솔(phpMyAdmin)이 암호화되지 않은 평문 프로토콜로 서비스되고 있음을 확인 |
| 2. 자격 증명 접근 | TA0006 Credential Access (T1557 Adversary-in-the-Middle, T1557.002 ARP Cache Poisoning) | ARP Cache Poisoning을 통해 내부망 호스트와 DB 서버 구간의 트래픽을 공격자 경유로 우회시키고, phpMyAdmin 세션 및 인증 정보를 가로챔 |
| 3. 실행 | TA0002 Execution (T1213 Data from Information Repositories) | 가로챈 세션/자격 증명을 이용해 phpMyAdmin에 접근, 데이터베이스에 직접 쿼리를 실행 |
| 4. 영향 | TA0040 Impact | DB 직접 접근을 통한 회원·거래 데이터 조회/변조, 관리자 계정 권한 필드 직접 수정을 통한 권한 상승 가능 |

**결과**: 내부망 통신이 평문으로 이루어지고 phpMyAdmin에 별도 인증·접근 IP 제한이 없어, ARP Spoofing만으로 데이터베이스 전체에 대한 통제권을 확보할 수 있음이 확인됨.

### 5.5 시나리오 5 — 소스코드 저장소 접근 및 백도어 삽입을 통한 공급망 공격 (관련 발견 항목: #14)

**개요**: 공격자가 개발망의 소스코드 저장소(Gitea) 및 CI/CD(Jenkins) 접근 권한을 확보한 뒤, 소스코드에 백도어를 삽입하여 정상 배포 절차를 통해 악성코드를 운영 환경까지 유포하는 시나리오임. 시나리오 3에서 확보한 자격 증명·권한과 직접 연계됨.

| 단계 | ATT&CK 전술/기법 | 수행 내용 |
|---|---|---|
| 1. 정찰 | TA0043 Reconnaissance (T1593 Search Open Websites/Domains) | 시나리오 3에서 확보한 내부 정보(자격 증명 메모 등)를 바탕으로 소스코드 저장소(Gitea) 주소 및 접근 정보를 식별 |
| 2. 초기 접근 | TA0001 Initial Access (T1078 Valid Accounts, T1199 Trusted Relationship) | 탈취한 자격 증명 또는 Jenkins-Gitea 연동 토큰을 이용해 소스코드 저장소에 접근 |
| 3. 지속성 확보 | TA0003 Persistence (T1195.001 Compromise Software Supply Chain) | 백엔드 소스코드의 인증/결제 로직 등에 공격자가 제어 가능한 백도어(예: 특정 파라미터 수신 시 관리자 권한 부여)를 삽입하여 커밋 |
| 4. 영향 | TA0040 Impact | Jenkins 빌드 파이프라인이 별도 검증 없이 해당 변경을 빌드·배포함에 따라, 백도어가 포함된 이미지가 운영 환경에 그대로 반영됨 |

**결과**: 코드 저장소에 대한 접근 제어 및 변경 이력 검증(리뷰/서명) 체계가 없을 경우, 단 1회의 악성 커밋만으로 전체 배포 파이프라인이 공급망 공격의 매개체가 될 수 있음이 확인됨.

## 6. 문제상황 및 해결과정

### 6.1 주요 취약점 요약

본 진단을 통해 도출된 주요 취약점은 SQL Injection, Race Condition 기반 결제 로직 무결성 손상, 웹서버 취약점을 통한 내부망 침투 및 관리자 권한 상승, 내·외부 ARP Spoofing, 소스코드 저장소 접근 및 백도어 삽입 5개 항목이며, 모두 위험도 Critical로 평가됨.

| # | 주요 취약점 | 잠재적 영향 | 위험도 | 관련 시나리오 / 세부항목 |
|---|---|---|---|---|
| 1 | SQL Injection | 고객 개인정보 데이터베이스 유출 가능 | Critical | 시나리오 2 / 6.2-① |
| 2 | Race Condition 기반 결제 로직 무결성 손상 | 결제 금액 및 거래 신뢰성 훼손 | Critical | 시나리오 1 / 6.2-⑥ |
| 3 | 웹서버 취약점을 통한 내부망 침투 및 관리자 권한 상승 | 내부 시스템 권한 탈취 가능 | Critical | 시나리오 3 / 6.2-③, ⑪, ⑫ |
| 4 | 내·외부 ARP Spoofing | phpMyAdmin 서버 접근 가능 | Critical | 시나리오 4 / 6.2-⑬ |
| 5 | 소스코드 저장소 접근 및 백도어 삽입 | 코드 무결성 및 배포 시스템 위협 | Critical | 시나리오 5 / 6.2-⑭ |

위 5개 취약점은 개인정보 유출, 거래 무결성 훼손, 내부 시스템 장악, 관리자 권한 탈취, 소프트웨어 공급망 공격으로 이어질 수 있는 심각한 보안 위협으로 판단됨.

### 6.2 세부 발견 항목 및 개선 조치 (위험도 순)

| # | 영역 | 발견 내용(문제상황) | ATT&CK 매핑(예시) | 위험도 | 개선 조치(해결과정) |
|---|---|---|---|---|---|
| 1 | 인증(Backend) | 로그인 시 이메일 파라미터가 검증 없이 쿼리에 그대로 사용되어 SQL Injection이 가능하고, 실시간 채팅 메시지 저장 쿼리 또한 사용자 입력을 문자열로 결합해 동일한 위험이 존재함 (시나리오 2 참조) | T1190 (Exploit Public-Facing Application) | Critical | 모든 사용자 입력에 대해 파라미터 바인딩/ORM 쿼리 빌더 사용으로 전환, 입력값 화이트리스트 검증을 추가함 |
| 2 | 커뮤니티(Backend) | 게시글 제목/본문에 특정 템플릿 구문이 포함될 경우 서버 측에서 동적 코드 평가가 수행되어 서버 측 코드 실행(SSTI)으로 이어질 수 있는 처리 로직이 발견됨 | T1190 → T1059 (Command and Scripting Interpreter) | Critical | 사용자 입력에 대한 동적 코드 평가(eval 기반 처리) 로직을 제거하고, 출력 인코딩 및 템플릿 렌더링을 분리함 |
| 3 | 파일 업로드/다운로드(Backend) | 다운로드 API의 파일 경로 파라미터에 대한 검증이 부족해 경로 탐색(Path Traversal)이 가능했고, 업로드된 특정 확장자 파일에 대해 서버 명령 실행을 허용하는 정적 파일 핸들러 로직이 발견됨 (시나리오 3 참조) | T1190, T1505 (Server Software Component) | Critical | 다운로드 경로를 화이트리스트/내부 식별자 기반으로 재설계하고, 업로드 파일의 실행 권한 제거 및 확장자·MIME 검증을 강화함 |
| 4 | 인증/세션(Backend) | JWT 서명 키가 기본값으로 하드코딩되어 있고, 일부 API에서 서명 검증 없이 토큰 페이로드만 신뢰(`decode`)하는 로직이 확인됨. 비밀번호는 취약한 해시 알고리즘(MD5)을 사용하며, 세션 쿠키에 `HttpOnly`/`Secure` 속성이 적용되지 않음 | T1078 (Valid Accounts), T1552 (Unsecured Credentials) | High | JWT 시크릿을 환경별 강력한 값으로 분리 관리하고, 모든 보호 라우트에서 서명 검증을 강제함. 비밀번호 해시를 bcrypt 등으로 전환하고, 쿠키 보안 속성(HttpOnly/Secure/SameSite)을 적용함 |
| 5 | 상품/거래 API(Backend) | 상품 수정/삭제 API에서 소유자 확인 없이 요청이 처리되어 권한 검증 누락(IDOR/Broken Access Control)이 발견되었고, 일부 엔드포인트는 요청 바디를 그대로 모델에 반영해 대량 할당(Mass Assignment) 위험이 존재함 | T1548 (Abuse Elevation Control Mechanism) | High | 리소스 소유자/역할 기반 접근제어(RBAC) 미들웨어를 적용하고, 업데이트 허용 필드를 화이트리스트화함 |
| 6 | 거래 처리(Backend) | 구매 처리 시 재고 확인과 크레딧 차감 사이에 동시성 제어가 없어 레이스 컨디션(Race Condition)으로 인한 이중 구매·잔액 변조 가능성이 발견됨 (시나리오 1 참조) | CWE-362 (Race Condition), TA0040 Impact | Critical | DB 트랜잭션 및 행 단위 잠금(SELECT ... FOR UPDATE)을 적용하고, 재고/크레딧 갱신을 단일 원자적 트랜잭션으로 재구성함 |
| 7 | 거래 처리(Backend) | 배송 정보 입력값 검증에 사용된 정규식이 특정 입력에서 ReDoS(정규식 서비스 거부)를 유발할 수 있는 패턴으로 작성됨 | T1499 (Endpoint DoS) | Medium | 정규식을 비백트래킹 패턴으로 재작성하고, 입력 길이 제한 및 처리 타임아웃을 적용함 |
| 8 | API 공통(Backend) | `/api/health` 및 각종 에러 응답에서 서버 플랫폼·Node 버전·메모리 사용량·스택 트레이스 등 민감 정보가 노출됨 (시나리오 2의 정찰 단계에서 활용됨) | T1592 (Gather Victim Host Information) | Medium | 운영 환경에서 상세 에러/시스템 정보 응답을 비활성화하고, 헬스체크 응답을 최소 정보로 축소함 |
| 9 | 인증(Backend) | 회원가입/비밀번호 재설정 API에서 이메일 존재 여부를 응답 메시지로 알 수 있는 사용자 열거(User Enumeration)와, 비밀번호 재설정 토큰이 응답 본문에 직접 포함되는 문제가 발견됨 | T1589 (Gather Victim Identity Information) | Medium | 응답 메시지를 일반화하여 계정 존재 여부를 비노출하고, 재설정 토큰은 이메일 채널로만 전달함 |
| 10 | 플랫폼 공통(Backend) | CSP·X-Frame-Options 등 주요 보안 헤더가 비활성화되어 있고, CORS가 모든 출처(`origin: true`)를 허용함 | T1185 (Browser Session Hijacking) 위험 증가 | Medium | CSP/Frame 관련 헤더를 활성화하고, CORS 허용 출처를 화이트리스트로 제한함 |
| 11 | 내부망 | 내부망에서 DB 관리 콘솔(phpMyAdmin)이 추가 인증 없이 접근 가능했고, 내부 업무 호스트에서 평문 자격증명·내부 인프라 정보가 포함된 메모 파일이 발견됨 (시나리오 3 참조) | T1083 (File and Directory Discovery), T1552 (Unsecured Credentials) | High | 관리 콘솔에 별도 인증/접근 IP 제한을 적용하고, 평문 자격증명 파일을 제거하며 비밀관리 솔루션 도입을 권고함 |
| 12 | 개발망(CI/CD) | Jenkins 컨테이너가 Docker 소켓에 접근 가능한 구조로, CI/CD 서버 침해 시 호스트 컨테이너 전체에 대한 권한 확대 가능성이 확인됨 (시나리오 3 참조) | T1610 (Deploy Container), T1611 (Escape to Host) | High | Docker 소켓 직접 마운트를 제거하고 격리된 빌드 에이전트(rootless/DinD) 사용을 권고함. 파이프라인 자격증명을 최소권한화함 |
| 13 | 내부망 | 내부망 통신이 암호화되지 않은 평문 프로토콜로 이루어져 ARP Spoofing(ARP Cache Poisoning)을 통한 트래픽 가로채기가 가능하며, 이를 통해 별도 인증이 없는 DB 관리 콘솔(phpMyAdmin)에 접근 가능함이 확인됨 (시나리오 4 참조) | T1557 (Adversary-in-the-Middle), T1557.002 (ARP Cache Poisoning) | Critical | 내부망 통신을 TLS 기반으로 전환하고, 동적 ARP 검사(DAI)·포트 시큐리티 등 L2 보안 기능을 적용함. 관리 콘솔에는 별도 인증 및 접근 IP 제한을 적용함 |
| 14 | 개발망(CI/CD) | 소스코드 저장소(Gitea) 및 CI/CD 연동 토큰에 대한 접근 제어가 미흡하여, 탈취한 자격 증명으로 저장소에 접근해 코드에 백도어를 삽입할 수 있는 가능성이 확인됨. 삽입된 코드는 Jenkins 빌드 파이프라인을 통해 검증 없이 그대로 배포됨 (시나리오 5 참조) | T1195.001 (Compromise Software Supply Chain), T1199 (Trusted Relationship) | Critical | 저장소 접근에 대한 다중 인증 및 접근 IP 제한을 적용하고, 커밋에 대한 코드 리뷰·서명 검증을 의무화함. 빌드 파이프라인에 정적분석(SAST)·이미지 스캔 단계를 추가하고, 접근·변경 이력에 대한 감사 로그 체계를 구축함 |

### 6.3 테스트베드 구축·검증 과정에서 발생한 이슈 및 해결

블랙박스 환경을 고객사 운영 환경과 동일하게 재현하는 과정에서 다음과 같은 운영상 이슈가 발생하였으며, 모두 해결 후 본 진단을 진행함.

- **실시간 채팅 환경 불안정**: Socket.IO 서버/클라이언트 간 CORS 및 채팅방 식별자 충돌로 메시지 송수신이 불안정 → 채팅방 식별자를 거래(상품) 식별자와 분리하고, CORS 허용 출처 및 통신 방식(polling/websocket)을 명시적으로 설정하여 해결함
- **인증 모듈 버전 호환성 문제**: 인증 토큰 라이브러리 버전 업그레이드 후 로그인 검증 로직과의 호환성 문제로 로그인 실패가 발생 → 호환 버전으로 조정하여 해결함. 이후 6.2-④ 항목으로 별도 진단을 진행함
- **DB 관리 콘솔 접근 불가**: 테스트베드 내 DB 관리 콘솔 컨테이너의 설정 오류로 접근이 불가 → 이미지/환경변수 재설정 후 접근 가능하도록 조치함 (이후 6.2-⑪, 6.2-⑬ 항목에서 접근 제어 및 네트워크 보안 미흡이 추가로 발견됨)
- **한글 데이터 인코딩 오류**: 데이터베이스 문자셋 설정 문제로 샘플 데이터의 한글이 깨져 표시됨 → UTF-8(utf8mb4) 설정 적용으로 해결함
- **포트 충돌**: 로컬/CI 환경에 따라 서비스 포트가 충돌하여 백엔드 기동이 실패함 → 포트 충돌 자동 점검 및 대안 포트 안내 로직을 적용하여 해결함

## 7. 개선 권고사항

발견된 취약점에 대한 개선 권고사항은 다음과 같이 우선순위를 설정하여 도출됨.

| 우선순위 | 영역 | 권고 내용 | 관련 항목 |
|---|---|---|---|
| 1 (최우선) | 결제 로직 개선 | Race Condition을 방지하고 트랜잭션 무결성을 강화함 | 6.2-⑥, 시나리오 1 |
| 2 | 웹 애플리케이션 보안 강화 | SQL Injection 취약점을 제거하고 입력값 검증을 강화함 | 6.2-①, 시나리오 2 |
| 3 | 웹서버 및 내부망 보안 개선 | 접근 통제를 강화하고 권한 관리 정책을 수립·적용함 | 6.2-③, ⑪, ⑫, 시나리오 3 |
| 4 | 네트워크 보안 강화 | ARP Spoofing 공격에 대한 모니터링 체계를 구축하고 내부망 세그멘테이션을 적용함 | 6.2-⑬, 시나리오 4 |
| 5 (마지막) | 코드 저장소 보안 강화 | 백도어 삽입 여부를 점검하고 접근 제어 및 감사 로그 체계를 강화함 | 6.2-⑭, 시나리오 5 |

이를 통해 결제 시스템의 무결성 확보, 웹 애플리케이션 취약점 제거, 내부 시스템 접근 통제 강화, 네트워크 공격 대응체계 구축, 소프트웨어 공급망 보안 강화를 단계적으로 수행할 수 있음.

## 8. 활용 방안

- **DFD/STRIDE 위협 테이블**: 데이터 흐름과 잠재적 위협을 식별·분류한 문서로, 향후 신규 기능 추가 시 위협 모델링 템플릿으로 재사용 가능함
- **ATT&CK 매핑 시나리오**: 공격 전술/기법 단위로 작성되어, 보안관제(SOC)의 탐지 룰 점검 및 모의훈련(Red Team) 시나리오로 재사용 가능함
- **결과 리포트**: 발견된 취약점, 위험도 평가, 개선 권고 및 대응 가이드를 포함하여 경영진 보고 및 기술팀 개선 작업의 기준 자료로 활용함

## 9. 기대 효과

- 시스템 구조와 위협을 한눈에 파악하고, 우선순위가 높은 취약점을 식별할 수 있음
- 실제 공격자 관점에서 보안 허점을 검증하여, 보안 정책·아키텍처 개선의 근거를 마련함
- Critical/High 항목에 대한 신속한 개선으로 사고 발생 전 예방 효과를 확보함
- 단계별 실행 가이드(개선 우선순위, 권고안)를 통해 보안 프로세스의 조직 내 정착을 지원함
